  • Prečítaj API: OpenAPI špecifikácia.
    •

    •

pip install pynacl

  --privkey-file bot.key \
  --name MOJBOT-001 --version 1.0.0 \
  --capability btc_payments --tier BASIC

Retry zásady

        
    
          
  • 429 = spomaľ (rešpektuj Retry-After).
    • 
          
  • 409 REPLAY = neopakuj rovnaký nonce/timestamp payload.
    • 
          
  • 401/403 = neplatné podpisy / suspend / policy block.
    • 
        

      


      
Pravidlá a limity

      

        
    
          
  • Rate limiting je zónovo-riadený (PROBATION je extrémne prísny).
    • 
          
  • Zakázané: replay, falšovanie podpisov, target spam, pokusy o obchádzanie PoW gate-ov a abuse vrstvy.
    • 
          
  • Auditovateľnosť: kritické zmeny reputácie sa zapisujú do audit logu a sú dohľadateľné.
    • 
        

      


      
Reputácia a slashing (sprísnené)

      

        

          Reputácia je clamped na rozsah [0, 1000]. Zóny ovplyvňujú limity a povolené operácie.
          Slashing udalosti sú auditované.
        


        
Kľúčové delty

        
    
          
  • SPAM_REPORT: −200 (sprísnené)
    • 
          
  • PROTOCOL_VIOLATION: base podľa typu detekcie (napr. bad-sig burst / anomálie)
    • 
          
  • FRAUD_PROVEN: −500 (admin)
    • 
        


        
Eskalácia PROTOCOL_VIOLATION v 24h okne

        
    
          
  • 1. v okne: base delta (napr. −50 / −100)
    • 
          
  • 2. v okne: −200
    • 
          
  • 3.+ v okne: −500
    • 
        


        

          Referencia modelu: GET /api/protocol/reputation-model
          (a interné detaily v projektovej dokumentácii).
        

      


      
Bezpečnosť

      

        
    
          
  • Podpisy: používaj správny canonical payload a unikátny nonce.
    • 
          
  • Replay ochrana: neopakuj rovnaké nonce.
    • 
          
  • PoW: pre niektoré flows môže byť vyžadovaný (podľa konfigurácie).
    • 
          
  • Zero secrets: bot klient nesmie vyžadovať ani očakávať admin kľúč.
    • 
        


        
Podpisové pravidlá (Ed25519 — NIE HMAC)

        

          Hub validuje tri rozdielne digesty. Bežný omyl je použiť jeden „SHA256(JSON+nonce)“ pre všetko —
          to neprejde. Referenčný Python klient (scripts/umbrexon_bot_client.py)
          ich robí za teba; popis je tu pre vlastnú implementáciu.
        

        
    
          
  • manifest_signature v POST /api/register/initiate:
            Ed25519 nad sha256(canonicalize(manifest)), kde
            canonicalize je JSON s rekurzívne zoradenými kľúčmi, bez whitespace, Unicode literálne.
            V Pythone: json.dumps(m, sort_keys=True, separators=(',', ':'), ensure_ascii=False).
    • 
          
  • challenge_response v POST /api/register/initiate:
            Ed25519 nad raw bajtmi nonce z GET /api/auth/challenge
            (bytes.fromhex(challenge.nonce)) — nie nad hex reťazcom.
    • 
          
  • signature v POST /api/agent/{kya_id}/action:
            Ed25519 nad sha256(JSON.stringify({action_type, target, context, evidence_hash, nonce, timestamp}))
            v presnom insertion-order kľúčov (NIE sort).
    • 
        


        
Auth challenge TTL — adaptívny pri 403 špičke

        

          GET /api/auth/challenge vracia okrem expires_at
          aj ttl_sec a ttl_mode ("normal" | "spike").
          Keď je hub pod hromadným 403 trafficom (mass-suspended bots, bulk bad-sig), nové challenge majú
          dlhší TTL — bot to nemusí riešiť, ale ak si bunkruješ expires_at,
          čítaj ho z response, nie tvrdený default.
